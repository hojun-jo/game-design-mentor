# 게임 기획 멘토 MVP 기획서

## 1. Product Summary

1차 타깃은 게임 개발이 익숙하지 않은 초보 인디 개발자다. 사용자는 형식이 완전히 고정되지 않은 마크다운 기획 초안을 준비하고, 파이썬 CLI에 한 번 넣어 `무엇을 먼저 정해야 하는지`를 묻는다. 핵심은 AI가 더 멋진 게임을 대신 만드는 것이 아니라 `의도`, `대상 플레이어`, `핵심 루프`, `MVP 범위`, `플레이테스트 질문`을 기준으로 사용자가 스스로 다음 결정을 내리게 돕는 것이다.

현재 MVP는 `파이썬 CLI 기반 일회성 리뷰 도구`다. 저장형 워크스페이스, 버전 히스토리, 플레이테스트 회고 누적보다 `질문 + 짧은 진단 + 2개의 해석 방향`이 실제로 쓸모 있는지 먼저 검증한다.

## 2. Target Users

- 1차 타깃: 게임 개발이 익숙하지 않은 초보 인디 개발자
- 후속 타깃 후보: 대학생 포트폴리오 제작자, 주니어 기획자
- 현재 MVP에서는 학생 발표용 흐름, B2B 표준화, 팀 회고 워크플로는 우선하지 않는다.

## 3. Problem Statement

초보 인디 개발자는 첫 기획에서 `무엇이 빠졌는지`, `어디가 모순인지`, `뭘 먼저 정해야 하는지`를 스스로 구분하기 어렵다. 창작 피드백은 자주 `좋다`, `별로다`, `더 참신해야 한다` 같은 감상으로 흐르고, 게임 기획에서는 특히 `누구를 위한 게임인지`, `핵심 루프가 정말 서는지`, `무엇을 빼야 하는지`, `나중에 뭘 테스트해야 하는지`가 흐려진다.

이 제품은 사용자의 취향을 대신 결정하지 않는다. 대신 질문과 짧은 진단, 그리고 트레이드오프가 붙은 2개의 방향 제시를 통해 사용자가 다음 결정을 내리게 돕는다.

## 4. MVP Goal

- 사용자는 첫 10분 안에 마크다운 기획 초안을 CLI에 넣고 1차 피드백을 받을 수 있다.
- AI는 최소 `대상 플레이어`, `의도한 감정`, `코어 루프`, `범위`, `플레이테스트 질문` 5영역을 점검해야 한다.
- 첫 응답은 반드시 `질문 + 짧은 진단` 구조를 가져야 한다.
- 리뷰는 정확히 2개의 해석 방향을 제시하고, 각 방향에 한 줄 근거와 한 줄 트레이드오프를 붙여야 한다.
- `코어 루프`, `감정 목표`가 비면 본 리뷰 대신 질문 중심 응답으로 전환해야 한다.
- 자유형 마크다운은 먼저 LLM이 표준 JSON 구조로 정리하고, 파이썬 코드는 그 JSON을 검증해야 한다.
- 초기 성공 신호는 사용자가 첫 피드백 이후 `무엇을 먼저 정해야 하는지 알겠다`고 느끼는 것이다.
- 다음 성공 신호는 사용자가 `기획안을 수정해서 다시 넣고 싶다`고 느끼는 것이다.

## 5. Core User Scenarios

1. 초보 인디 개발자는 로그라이크 초안을 넣고 `누구를 위한 게임인지`, `루프가 너무 복잡한지`, `무엇을 먼저 빼야 하는지`를 질문받는다.
2. 초보 인디 개발자는 퍼즐 게임 기획을 넣고 `감정 목표와 루프가 맞는지`, `첫 테스트에서 뭘 물어봐야 하는지`를 짧게 진단받는다.
3. 사용자는 하나의 기획 초안에 대해 2개의 해석 방향과 각각의 트레이드오프를 비교받고, 지금 어떤 선택이 더 중요한지 확인한다.

## 6. Detailed MVP Feature List

### Feature 1. 기획 의도와 대상 플레이어 정렬 진단

**Why it matters**

좋은 게임 기획 피드백은 메커닉보다 먼저 `누구를 위해 어떤 감정을 만들려는가`를 확인한다.

**Main user flow**

1. 사용자가 게임 제목, 장르, 대상 플레이어, 의도한 감정, 한 줄 콘셉트를 입력한다.
2. AI가 누락 정보와 모순을 찾는다.
3. 대상 플레이어가 비어 있어도 리뷰는 진행하되, 첫 질문에서 이를 우선 확인한다.

**Inputs and outputs**

- 입력: 게임 콘셉트, 대상 플레이어, 감정 목표, 레퍼런스, 한 줄 설명
- 출력: 누락 질문, 정렬 불일치 진단, 수정 제안

**Edge cases or failure cases**

- 대상 플레이어가 너무 넓으면 피드백이 공허해질 수 있다.
- 레퍼런스만 많고 자기 의도가 없으면 판단 기준이 흔들린다.

**Acceptance criteria**

- 대상 플레이어, 감정 목표, 콘셉트의 연결 여부를 점검해야 한다.
- 누락 정보가 있으면 먼저 질문을 반환해야 한다.
- 대상 플레이어가 비어 있으면 첫 질문에서 이를 가장 먼저 확인해야 한다.

### Feature 2. 코어 루프와 차별화 포인트 리뷰

**Why it matters**

아이디어가 있어도 반복 플레이 가치가 없으면 게임이 서지 않는다. 코어 루프 검토가 핵심이다.

**Main user flow**

1. 사용자가 핵심 루프와 주요 시스템을 입력한다.
2. AI가 입력, 보상, 반복 동기, 난이도 상승의 연결을 본다.
3. AI가 짧은 진단과 2개의 해석 방향을 보여준다.

**Inputs and outputs**

- 입력: 코어 루프, 주요 시스템, 보상 구조, 차별화 포인트
- 출력: 루프 약점 진단, 반복성 피드백, 2개의 해석 방향

**Edge cases or failure cases**

- 기능 나열만 있고 루프 연결이 없으면 안 된다.
- 차별화 포인트가 시장 문구 수준이면 안 된다.

**Acceptance criteria**

- 코어 루프의 시작, 반복, 보상 구조를 분리해 검토할 수 있어야 한다.
- 차별화 포인트가 실제 플레이 경험과 연결되는지 설명해야 한다.
- 2개의 해석 방향과 각 방향의 근거, 트레이드오프를 보여줄 수 있어야 한다.

### Feature 3. MVP 범위와 플레이테스트 계획 점검

**Why it matters**

좋은 기획도 범위가 터지면 못 만든다. 무엇을 먼저 검증할지 정리해야 한다.

**Main user flow**

1. 사용자가 기간, 인원, 기능 목록을 입력한다.
2. AI가 과한 범위와 빠진 검증 단계를 지적한다.
3. AI가 지금 빼야 할 것과 첫 테스트 질문을 짧게 정리한다.

**Inputs and outputs**

- 입력: 개발 기간, 인원, 기능 목록, MVP 목표, 테스트 대상
- 출력: 범위 축소 제안, 우선 검증 항목, 플레이테스트 질문

**Edge cases or failure cases**

- 범위만 줄이고 핵심 재미 검증 목표가 없으면 안 된다.
- 테스트 계획이 모호하면 회고도 약해진다.

**Acceptance criteria**

- 최소 1개의 핵심 검증 가설과 2개의 테스트 질문을 도출해야 한다.
- 현재 범위에서 제외할 항목을 제안할 수 있어야 한다.
- 일정이나 인원 정보가 부족해도 보수적 가정 위에서 임시 범위 제안을 제공할 수 있어야 한다.

### Feature 4. 플레이테스트 회고와 다음 버전 제안

**Why it matters**

장기적으로는 중요한 확장 기능이지만, 현재 첫 릴리스의 핵심 검증 범위는 아니다.

**Main user flow**

1. 이 기능은 첫 릴리스 이후 단계에서 검토한다.
2. 초기 MVP에서는 플레이테스트 질문 생성까지 제공한다.
3. 실제 회고 누적과 다음 버전 제안은 후속 단계로 분리한다.

**Inputs and outputs**

- 입력: 테스트 메모, 이탈 지점, 플레이어 반응, 버전 변경 기록
- 출력: 회고 요약, 원인 후보, 다음 버전 수정안

**Acceptance criteria**

- 현재 첫 릴리스 범위에는 포함하지 않는다.
- 질문 중심 리뷰 품질이 검증된 뒤 다시 포함 여부를 평가한다.

## 7. Why This Fits Solo Development

- 강의 콘텐츠를 만드는 대신 `구조화된 기획 입력 + 질문 중심 리뷰 로직`에 집중하면 범위를 줄일 수 있다.
- 게임 기획 문서는 텍스트 중심이라 복잡한 미디어 처리 없이도 MVP 검증이 가능하다.
- 리뷰 품질이 먼저 검증되면 이후에 학생용, 포트폴리오용, 팀용으로 확장하기 쉽다.
- 플레이테스트 회고와 버전 비교는 리뷰 엔진이 검증된 뒤 붙여도 늦지 않다.

- 나쁜 코어 루프 예시와 개선 예시 생성
- 플레이테스트 질문 템플릿 생성
- 2개의 해석 방향과 트레이드오프 초안 생성

- 플레이테스트 회고 누적과 다음 버전 기록 관리

- 플레이 영상 자동 분석

## 10. Confirmed MVP Decisions and Assumptions

### Assumptions

- 1차 플랫폼은 파이썬 CLI다.
- 사용자는 텍스트 중심의 마크다운 파일로 기획 초안을 입력한다.
- 마크다운 파일 형식은 느슨하게 두고, 구조화는 LLM extraction이 맡는다.
- 파이썬 코드는 자유형 텍스트를 직접 해석하기보다 LLM이 만든 구조화 결과를 검증한다.
- LLM은 추출과 질문/피드백 생성에 쓰고, 점검 체크리스트는 앱 코드가 가진다.
- 피드백 기준은 사용자의 목표, 대상, 핵심 루프 위에 세운다.

### Confirmed MVP Decisions

- 범용 창작 피드백이 아니라 초보 인디 개발자용 게임 기획 멘토로 좁힌다.
- 첫 화면 질문은 항상 `어떤 플레이어에게 어떤 감정을 주고 싶나요?`로 시작한다.
- 첫 릴리스는 웹 앱보다 파이썬 CLI + 마크다운 입력 파일로 시작한다.
- 첫 릴리스는 저장형 워크스페이스가 아니라 일회성 리뷰 도구로 시작한다.
- 첫 릴리스는 멀티에이전트 분해보다 `extractor -> validator -> reviewer` 3단계 흐름으로 시작한다.
- 피드백은 `질문`, `짧은 진단`, `2개의 해석 방향`, `트레이드오프` 구조로 준다.
- `코어 루프`, `감정 목표`가 없으면 본 리뷰 대신 질문 중심 응답으로 전환한다.
- `대상 플레이어`는 권장 필드로 두되, 비어 있으면 첫 질문에서 우선 확인한다.
- AI는 정답을 단정하지 않고 `2개의 선택지와 트레이드오프`를 먼저 보여준다.

## 11. Shared Product Structure

### 11.1 Core Session Loop

- 게임 기획 의도 입력
- 대상 플레이어와 감정 목표 정렬 검토
- 코어 루프와 차별화 리뷰
- MVP 범위와 테스트 계획 점검
- 2개의 해석 방향과 트레이드오프 비교
- 다음 우선 결정 하나를 명시하고 종료

### 11.2 Product States

- `Concept Intake`: 콘셉트와 의도 입력
- `Clarify Missing`: 핵심 정보 누락 질문
- `Mentor Review`: 정렬 불일치와 약점 진단
- `Decision Compare`: 2개의 해석 방향과 트레이드오프 비교
- `Design Plan`: 코어 루프와 MVP 범위, 테스트 질문 정리

## 12. Good Mentor Pattern

- 먼저 고쳐주지 말고 `무엇을 지키려는 기획인지`를 묻는다.
- 문제를 말할 때는 관찰과 해석을 분리한다.
- 항상 2개의 대안을 주고, 각 대안의 장점과 비용을 같이 보여준다.
- 추천 대신 `지금 우선하는 것은 명확성, 참신함, 제작 가능성 중 무엇인가요?` 같은 선택 질문으로 끝낸다.
- 첫 릴리스에서는 회고를 누적하지 않고, 지금 무엇을 먼저 정해야 하는지 한 줄로 끝낸다.
