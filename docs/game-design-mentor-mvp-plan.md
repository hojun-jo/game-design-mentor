# 게임 기획 멘토 MVP 기획서

## 1. Product Summary

1차 타깃은 게임 개발이 익숙하지 않은 초보 인디 개발자다. 사용자는 형식이 완전히 고정되지 않은 기획 초안을 웹 앱에 붙여넣거나 마크다운 파일로 올리고, `무엇을 먼저 정해야 하는지`를 묻는다. 핵심은 AI가 더 멋진 게임을 대신 만드는 것이 아니라 `의도`, `대상 플레이어`, `핵심 루프`, `MVP 범위`, `플레이테스트 질문`을 기준으로 사용자가 스스로 다음 결정을 내리게 돕는 것이다.

현재 MVP는 `웹 기반 일회성 리뷰 도구`다. 저장형 워크스페이스, 버전 히스토리, 플레이테스트 회고 누적보다 `질문 + 짧은 진단 + 2개의 해석 방향`이 실제로 쓸모 있는지 먼저 검증한다.

## 2. Target Users

- 1차 타깃: 게임 개발이 익숙하지 않은 초보 인디 개발자
- 후속 타깃 후보: 대학생 포트폴리오 제작자, 주니어 기획자
- 현재 MVP에서는 학생 발표용 흐름, B2B 표준화, 팀 회고 워크플로는 우선하지 않는다.

## 3. Problem Statement

초보 인디 개발자는 첫 기획에서 `무엇이 빠졌는지`, `어디가 모순인지`, `뭘 먼저 정해야 하는지`를 스스로 구분하기 어렵다. 창작 피드백은 자주 `좋다`, `별로다`, `더 참신해야 한다` 같은 감상으로 흐르고, 게임 기획에서는 특히 `누구를 위한 게임인지`, `핵심 루프가 정말 서는지`, `무엇을 빼야 하는지`, `나중에 뭘 테스트해야 하는지`가 흐려진다.

이 제품은 사용자의 취향을 대신 결정하지 않는다. 대신 질문과 짧은 진단, 그리고 트레이드오프가 붙은 2개의 방향 제시를 통해 사용자가 다음 결정을 내리게 돕는다.

## 4. MVP Goal

- 사용자는 첫 10분 안에 웹 앱에 기획 초안을 넣고 1차 피드백을 받을 수 있다.
- AI는 최소 `대상 플레이어`, `의도한 감정`, `코어 루프`, `범위`, `플레이테스트 질문` 5영역을 점검해야 한다.
- 첫 응답은 반드시 `질문 + 짧은 진단` 구조를 가져야 한다.
- 리뷰는 정확히 2개의 해석 방향을 제시하고, 각 방향에 한 줄 근거와 한 줄 트레이드오프를 붙여야 한다.
- `코어 루프`, `감정 목표`가 비면 본 리뷰 대신 질문 중심 응답으로 전환해야 한다.
- 질문에 대한 사용자 답변을 반영한 뒤 다시 리뷰 흐름으로 복귀할 수 있어야 한다.
- 자유형 초안은 먼저 LLM이 표준 JSON 구조로 정리하고, 파이썬 코드는 그 JSON을 검증해야 한다.
- 첫 릴리스 MVP는 `구조화 -> 검증 -> 질문 보완 -> 3개 리뷰 축 -> 방향 비교 -> 최종 출력` 전체 흐름까지 포함한다.
- 초기 성공 신호는 사용자가 첫 피드백 이후 `무엇을 먼저 정해야 하는지 알겠다`고 느끼는 것이다.
- 다음 성공 신호는 사용자가 `기획안을 수정해서 다시 넣고 싶다`고 느끼는 것이다.

## 5. Core User Scenarios

1. 초보 인디 개발자는 로그라이크 초안을 웹 앱에 붙여넣고 `누구를 위한 게임인지`, `루프가 너무 복잡한지`, `무엇을 먼저 빼야 하는지`를 질문받는다.
2. 초보 인디 개발자는 퍼즐 게임 기획 파일을 업로드하고 `감정 목표와 루프가 맞는지`, `첫 테스트에서 뭘 물어봐야 하는지`를 짧게 진단받는다.
3. 사용자는 하나의 기획 초안에 대해 2개의 해석 방향과 각각의 트레이드오프를 비교받고, 지금 어떤 선택이 더 중요한지 확인한다.

## 6. Detailed MVP Feature List

### Feature 1. 기획 의도와 대상 플레이어 정렬 진단

**Why it matters**

좋은 게임 기획 피드백은 메커닉보다 먼저 `누구를 위해 어떤 감정을 만들려는가`를 확인한다.

**Main user flow**

1. 사용자가 게임 제목, 장르, 대상 플레이어, 의도한 감정, 한 줄 콘셉트를 입력한다.
2. AI가 누락 정보와 모순을 찾는다.
3. 대상 플레이어가 비어 있어도 리뷰는 진행하되, 첫 질문에서 이를 우선 확인한다.

**Inputs and outputs**

- 입력: 게임 콘셉트, 대상 플레이어, 감정 목표, 레퍼런스, 한 줄 설명
- 출력: 누락 질문, 정렬 불일치 진단, 수정 제안

### Feature 2. 코어 루프와 차별화 포인트 리뷰

**Why it matters**

아이디어가 있어도 반복 플레이 가치가 없으면 게임이 서지 않는다. 코어 루프 검토가 핵심이다.

**Inputs and outputs**

- 입력: 코어 루프, 주요 시스템, 보상 구조, 차별화 포인트
- 출력: 루프 약점 진단, 반복성 피드백, 2개의 해석 방향

### Feature 3. MVP 범위와 플레이테스트 계획 점검

**Why it matters**

좋은 기획도 범위가 터지면 못 만든다. 무엇을 먼저 검증할지 정리해야 한다.

**Inputs and outputs**

- 입력: 개발 기간, 인원, 기능 목록, MVP 목표, 테스트 대상
- 출력: 범위 축소 제안, 우선 검증 항목, 플레이테스트 질문

### Feature 4. 플레이테스트 회고와 다음 버전 제안

후속 단계 기능이다. 첫 릴리스에는 포함하지 않는다.

## 7. Out-of-Scope Items

- 저장형 워크스페이스와 버전 히스토리
- 플레이테스트 회고 누적과 다음 버전 기록 관리
- 실시간 팀 공동 편집
- 플레이 영상 자동 분석
- 완전 자동 게임 기획 생성기

## 8. Confirmed MVP Decisions and Assumptions

### Assumptions

- 1차 플랫폼은 파이썬 기반 웹 앱이다.
- 사용자는 텍스트 붙여넣기 또는 마크다운 파일 업로드로 기획 초안을 넣는다.
- 입력 형식은 느슨하게 두고, 구조화는 LLM extraction이 맡는다.
- 파이썬 코드는 자유형 텍스트를 직접 해석하기보다 LLM이 만든 구조화 결과를 검증한다.
- 첫 릴리스는 저장형 워크스페이스가 아니라 일회성 리뷰 도구다.

### Confirmed MVP Decisions

- 범용 창작 피드백이 아니라 초보 인디 개발자용 게임 기획 멘토로 좁힌다.
- 첫 화면 질문은 항상 `어떤 플레이어에게 어떤 감정을 주고 싶나요?`로 시작한다.
- 첫 릴리스는 웹 앱으로 시작한다.
- 첫 릴리스는 `extractor -> validator -> clarify -> intent review -> core loop review -> scope/playtest plan review -> direction compare -> final output` 전체 흐름을 구현한다.
- `코어 루프`, `감정 목표`가 없으면 본 리뷰 대신 질문 중심 응답으로 전환한다.
- 질문으로 보완된 답변은 다시 구조화/검증한 뒤 본 리뷰로 연결한다.
- `대상 플레이어`는 권장 필드로 두되, 비어 있으면 첫 질문에서 우선 확인한다.
- `플레이테스트 회고`는 MVP 범위가 아니라, `플레이테스트 계획 생성`까지만 MVP에 포함한다.

## 9. Shared Product Structure

- `Concept Intake`: 콘셉트와 의도 입력
- `Clarify Missing`: 핵심 정보 누락 질문
- `Mentor Review`: 의도 정렬, 코어 루프, 범위/테스트 계획 리뷰
- `Decision Compare`: 2개의 해석 방향 비교
- `Design Plan`: MVP 범위 축소와 플레이테스트 계획 정리

## 10. MVP Internal Data Contract

최종 MVP 흐름을 안정적으로 실행하려면 내부 상태도 공통 shape를 가져야 한다.

### Structured brief fields

- `concept_statement`: 게임 콘셉트 한 문장
- `target_player`: 핵심 타깃 플레이어
- `emotion_goal`: 의도한 감정
- `core_loop`: 반복 행동 흐름 요약
- `feature_list[]`: 주요 기능 목록
- `reward_structure`: 플레이어가 반복하게 되는 보상 구조
- `differentiation_points[]`: 차별화 포인트
- `development_window_weeks`: 개발 기간 주 단위
- `team_composition`: 인원과 역할 요약
- `mvp_goal`: 이번 MVP가 증명하려는 핵심 가설
- `test_audience`: 플레이테스트 대상
- `reference_titles[]`: 비교 기준으로 삼는 레퍼런스

### Validation classes

- `hard_required`: `emotion_goal`, `core_loop`
- `soft_required`: `target_player`, `reward_structure`, `mvp_goal`
- `optional`: `reference_titles`, `development_window_weeks`, `team_composition`, `test_audience`, `differentiation_points`

### Review output fields

- `intent_diagnosis`: 대상, 감정, 콘셉트 정렬 진단
- `core_loop_diagnosis`: 반복 구조와 보상 구조 진단
- `scope_diagnosis`: 현재 범위 현실성 진단
- `scope_recommendations[]`: 제외/축소 권고안
- `playtest_hypothesis`: 가장 먼저 검증할 가설
- `playtest_questions[]`: 최소 2개의 테스트 질문
- `direction_options[2]`: 각 방향의 제목, 근거, 트레이드오프
- `final_summary`: 지금 먼저 정해야 할 것 한 줄

## 11. Good Mentor Pattern

- 먼저 고쳐주지 말고 `무엇을 지키려는 기획인지`를 묻는다.
- 문제를 말할 때는 관찰과 해석을 분리한다.
- 항상 2개의 대안을 주고, 각 대안의 장점과 비용을 같이 보여준다.
- 첫 릴리스에서는 회고를 누적하지 않고, 지금 무엇을 먼저 정해야 하는지 한 줄로 끝낸다.
